About:
What is this?

FITeagle is a semantic-based open source resource management framework for federated infrastructures.

Within the context of Future Internet research activities different approaches are in the process of being developed. Starting with incremental evolutionary attempts onwards to complete revolutionary ideas.

In order to evaluate new protocols and architectures testbeds with a wide range of heterogeneous resources and variability in terms of size and complexity are needed.

Although there are several existing wireless and wired testbeds to run experiments on, they are - in terms of user management, experiment control, connectivity, ... - independent of each other.

FITeagle provides an extensible framework to seamlessly interconnect and control experiments with arbitrary resources in a wide range of available testbeds. This includes the interoperability with other federated testbed management and experimentation frameworks.

FITeagle is mainly based on the FIRE-Teagle developments and therefore inherits its basic conceptional architecture. FIRE-Teagle was successfully deployed and used in the Panlab project and is currently going to be extended within the OpenLab and Fed4FIRE project.

Extensibility:
Add resources

Any kind of resources can be easily integrated with FITealge just by implementing a new resource adapter.

A resource adapter represents an interface between a resource itself and FITeagle enabling experimenters using FITeagle to access, use and manage available resources.

Implementing a new resource adapter is quite simple, because all adapters’ implementations follow certain steps and just few methods should be implemented reflecting specific functionalities for each resource. Here is information about how to write your own research adapters...

Open-Multinet support:
Try it out

At the heart of the FITeagle framework is the Open Multinet information model. Built upon a solid base of earlier research in semantic network description, it is used to describe resources, including all aspects of the resource lifecycle. Drawing upon the well-established semantic web stack of technologies, RDF triple stores are built up to describe any existing or desired resource. In this way, information about a resource can be easily extracted, separating it from the concrete syntax of a given resource description, thus increasing the scalability and extensibility of the framework.

For legacy and interoperability purposes, the Open Multinet Java API provides a translator. The API's web service, built on REST principles, can translate from Open Multinet to GENI RSpec or OASIS TOSCA and back. You can currently access a running version of the Open Multinet Java API by sending a request to the web service located at http://demo.fiteagle.org:8080/omnlib/convert/, along with the location of a local file, the type of RSpec and the type of output file desired. For example:

curl --data-urlencode content@src/test/resources/geni/request/request_unbound.xml http://demo.fiteagle.org:8080/omnlib/convert/request/ttl

For more information, see our documentation over at open-multinet.info.
